# Review Findings

1. High: The watchlist delete flow now violates the new `positions ∪ watchlist` invariant and would drop live pricing for held positions. In [planning/MARKET_INTERFACE.md](/Users/sanamjan/LLMs_projects/Claude_Projects/finally/planning/MARKET_INTERFACE.md:497), `DELETE /api/watchlist/{ticker}` always calls `remove_ticker()`, but the design decision in [planning/MARKET_INTERFACE.md](/Users/sanamjan/LLMs_projects/Claude_Projects/finally/planning/MARKET_INTERFACE.md:507) says held portfolio tickers must remain tracked even after the user removes them from the watchlist. As written, removing a watched ticker that is still in `positions` would stop polling it and break portfolio P&L updates until some later code path re-adds it.

2. Medium: The startup and interface sections still describe the old watchlist-only ownership model, so the new tracking rule is not restored consistently after restart. The `MarketDataSource.start()` contract says implementations should watch for watchlist changes by re-reading cache or database in [planning/MARKET_INTERFACE.md](/Users/sanamjan/LLMs_projects/Claude_Projects/finally/planning/MARKET_INTERFACE.md:137), while the design decision later says ticker membership is now entirely the caller's runtime responsibility in [planning/MARKET_INTERFACE.md](/Users/sanamjan/LLMs_projects/Claude_Projects/finally/planning/MARKET_INTERFACE.md:507). The startup example also seeds `source.start()` from `get_default_watchlist() or DEFAULT_TICKERS` only in [planning/MARKET_INTERFACE.md](/Users/sanamjan/LLMs_projects/Claude_Projects/finally/planning/MARKET_INTERFACE.md:425), so held-but-unwatched positions still would not be tracked on boot.

Medium: The Massive API reference now presents `lastTrade["p"]` as the direct, canonical price in examples that are supposed to apply generally, but the interface doc correctly treats that field as plan-dependent. In [planning/MASSIVE_API.md](/Users/sanamjan/LLMs_projects/Claude_Projects/finally/planning/MASSIVE_API.md:93) and [planning/MASSIVE_API.md](/Users/sanamjan/LLMs_projects/Claude_Projects/finally/planning/MASSIVE_API.md:167), the examples index `item["lastTrade"]["p"]` without fallback. But [planning/MARKET_INTERFACE.md](/Users/sanamjan/LLMs_projects/Claude_Projects/finally/planning/MARKET_INTERFACE.md:277) explicitly notes `lastTrade.p` is only available on higher plans and falls back to `day.c`. Someone implementing from `MASSIVE_API.md` alone would either build code that throws when `lastTrade` is absent or conclude the free/delayed tiers support a field the rest of the design already treats as optional.

No other concrete regressions stood out in the current diff beyond those planning inconsistencies.
